# EddaCraft

**The forge where governed systems are shaped.**

EddaCraft is where tools for governing AI-assisted work are forged.  
It exists to introduce **memory, accountability, and restraint** into systems where automation would otherwise outrun understanding.

This is not an execution platform. 
It is first and foremost a governance platform.


## At a glance

**EddaCraft exists to help teams:**

- govern AI-assisted and automated work
- preserve durable context across repos, agents, and time
- explain *why* something happened, not just *what*
- slow down safely when the cost of mistakes is high

**Within EddaCraft live governed instruments such as:**

- **Anvil** — where work is shaped, constrained, and planned before execution  
- **Edda** — where accepted outcomes are preserved as immutable memory  

These are products that will be launching soon.

Alongside them, EddaCraft maintains **open-source instruments** that define shared language, structure, and constraints.

> EddaCraft governs.  
> Tools act within constraint.  
> Memory persists.


## What EddaCraft is

EddaCraft is not the hammer.  
It is not the spark.  
It is the place where both are permitted to exist **under constraint**.

Automation, agents, and harnesses may exist, now or in the future, but only to enforce intent, never to bypass it.

Authority precedes action.


## Why EddaCraft exists

The Eddas were not instruction manuals.  
They were records: stories, symbols, and decisions preserved so meaning could outlast the moment.

Modern software, especially AI-assisted software, moves faster than its ability to explain itself.

EddaCraft exists to slow the system just enough to answer:

- Why was this done?
- Who decided it?
- Under what constraints?

This is not nostalgia.  
It is discipline.


## The forge, redefined

EddaCraft is a forge... but not a chaotic one!

It is a controlled environment where:

- intent is shaped before action
- heat is applied deliberately
- every strike leaves a record

Ideas are not churned here.  
They are examined, constrained, and only then allowed to harden into form.

The forge stays quiet so the work stays legible.


## Core principles

### Governance first

EddaCraft’s centre of gravity is decision-making, not execution.

Systems may act quickly, but only after intent has been declared, constrained, and recorded.


### Memory before motion

Work is planned, reasoned about, and preserved before it is accelerated.

Context is not a side effect; it is a prerequisite.  
Speed is permitted only where memory survives.

If something cannot be explained later, it was not ready to happen sooner.


### Provenance over velocity

Outputs matter less than the ability to explain how they came to be.

EddaCraft optimises for systems that can answer **why**, not just **what**.

**If velocity obscures lineage, velocity loses.**


## Open source at EddaCraft

Some parts of the EddaCraft ecosystem are developed in the open.

These open-source instruments exist to establish **shared grammar and constraints**, not to act as systems of record.

Current open-source projects include:

- **Kindling** — capture, structure, and summarise working context before it hardens into memory  
- **Anvil Plan Spec** — a declarative specification for planning, intent, and constraint in AI-assisted work  

Additional open-source instruments will emerge over time where shared standards strengthen the ecosystem.

Authoritative memory, promotion, and enforcement remain within EddaCraft products.


## What EddaCraft is not

To set expectations clearly, EddaCraft is not:

- a general-purpose AI coding assistant
- a free-form agent playground
- an ungoverned automation engine

These boundaries are intentional.  
They may evolve, but the posture does not.

EddaCraft is designed to make unsafe speed uncomfortable.


## Who EddaCraft is for

EddaCraft is shaped for teams that:

- use or plan to use AI agents in production workflows
- require durable context across repos, agents, and time
- care about auditability, explainability, and institutional memory
- prefer systems that can slow down when it matters

It is particularly well-suited to environments where mistakes outlive deployments.
